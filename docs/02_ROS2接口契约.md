# 02 ROS2 接口契约

本文定义 Navigation 封装层必须长期保持稳定的 ROS2 topic、消息类型和语义。BotDog 后端可以直接继续使用这些接口。

## 坐标系

| 名称 | 默认值 | 说明 |
|---|---|---|
| 地图坐标系 | `map` | PCD、路径、导航点统一使用 |
| 机器人底盘坐标系 | `base_footprint` | BotDog 当前默认查询该 frame |
| 可选机体坐标系 | `base_link` | Super-LIO 当前会发布 `map -> base_link` |

封装层需要保证至少存在：

```text
map -> base_footprint
```

如果算法只输出 `map -> base_link`，需要通过静态 TF 或桥接节点补齐 `base_link -> base_footprint`。

## BotDog -> Navigation

| Topic | Type | 当前用途 | 保留要求 |
|---|---|---|---|
| `/clicked_point` | `geometry_msgs/msg/PointStamped` | 单点目标 | 必须保留 |
| `goal_yaw` | `std_msgs/msg/Float64` | 目标朝向 | 必须保留 |
| `/nav_start` | `std_msgs/msg/Bool` | 任务导航启动/停止 | 必须保留 |
| `/nav_stop` | `std_msgs/msg/Bool` | 急停/停止 | 必须保留 |
| `/initialpose` | `geometry_msgs/msg/PoseWithCovarianceStamped` | 重定位初始位姿 | 必须保留 |
| `/cmd_vel` | `geometry_msgs/msg/Twist` | 底盘速度桥输入 | 保留或明确替代 |

## Navigation -> BotDog

| Topic | Type | 当前用途 | 保留要求 |
|---|---|---|---|
| `/global_path` | `nav_msgs/msg/Path` | 全局路径展示/控制输入 | 必须保留 |
| `/nav_status` | `std_msgs/msg/String` JSON | 导航状态闭环 | 必须保留 |
| `/lio/cloud_world` | `sensor_msgs/msg/PointCloud2` | 建图预览 | 必须保留或 remap |
| `/lio/odom` | `nav_msgs/msg/Odometry` | 定位状态 | 建议保留 |
| `/weighted_ground` | `sensor_msgs/msg/PointCloud2` | planner ready 判断 | 建议保留 |
| TF `map -> base_footprint` | TF | 机器人位姿 | 必须稳定 |

## `/nav_status` JSON

推荐保持 BotDog 当前约定：

```json
{
  "status": "moving",
  "task_id": "task_001",
  "waypoint_id": "wp_001",
  "message": "导航中",
  "distance_to_goal": 1.25,
  "error_code": null,
  "timestamp": 1770000000.123
}
```

允许状态：

- `accepted`
- `moving`
- `reached`
- `failed`
- `canceled`
- `estop`

## 可新增但不强制的接口

后续为了跨进程、跨 Docker、跨机器更稳定，可以增加 JSON topic：

| Topic | Type | 说明 |
|---|---|---|
| `/nav_goal_json` | `std_msgs/msg/String` JSON | 替代 `/clicked_point + goal_yaw` 的扩展目标 |
| `/initialpose_json` | `std_msgs/msg/String` JSON | 跨系统发布重定位位姿 |
| `/nav_command_json` | `std_msgs/msg/String` JSON | 统一 start/stop/load_map 等命令 |

但第一阶段不依赖这些新增接口，避免 BotDog 大改。

## 运行时封装接口

`nav_runtime` 提供一层低耦合 topic 调度入口，用于让第三方后端通过 ROS2 topic 调用 Navigation 兼容脚本，而不是直接依赖脚本路径。

| Topic | Type | 方向 | 说明 |
|---|---|---|---|
| `/nav/command_json` | `std_msgs/msg/String` JSON | 上层 -> Navigation | 推荐命令入口 |
| `/nav_command_json` | `std_msgs/msg/String` JSON | 上层 -> Navigation | 兼容旧命名入口 |
| `/nav/runtime_status` | `std_msgs/msg/String` JSON | Navigation -> 上层 | 建图、导航、ready 文件、PID 状态汇总 |
| `/nav/command_result` | `std_msgs/msg/String` JSON | Navigation -> 上层 | 命令接收和启动结果 |

命令示例：

```json
{"command":"start_mapping","map_dir":"/home/frankluo/Projects/Navigation/maps/test_scene"}
```

```json
{"command":"restart_navigation","scene_dir":"/home/frankluo/Projects/Navigation/maps/test_scene"}
```

```json
{"command":"stop"}
```

`nav_runtime` 只负责调度和状态汇总，不直接依赖 BotDog 内部 API，不替代底层 SLAM、重定位和规划节点。

## 动态避障相关接口

动态避障第一阶段不改变全局规划接口，但应预留这些公共 topic：

| Topic | Type | 方向 | 说明 |
|---|---|---|---|
| `/nav/local_obstacles` | `sensor_msgs/msg/PointCloud2` | Navigation -> 上层/控制 | 机器人附近动态障碍点云 |
| `/nav/local_traversable` | `sensor_msgs/msg/PointCloud2` | Navigation -> 上层/控制 | 机器人附近实时可通行点云 |
| `/nav/obstacle_status` | `std_msgs/msg/String` JSON | Navigation -> 上层 | 动态避障状态、阻塞原因、最近障碍距离 |
| `/nav/replan_request` | `std_msgs/msg/Bool` | Navigation 内部/上层 | 动态障碍阻断路径时请求全局重规划 |
| `/cmd_vel_safe` | `geometry_msgs/msg/Twist` | Navigation -> 控制桥 | 经过避障限幅后的速度命令，可选 |

兼容当前系统时，可以继续使用现有内部 topic：

```text
/lio/cloud_world
/terrain_map
/base_footprint_fill_cloud
/local_map_raw
```

但对外文档应优先描述 `/nav/...` 公共接口。

## 规划 Action 兼容接口

`dddmr_global_planner-main` 里已有 `dddmr_sys_core/action/GetPlan` 风格的 action 接口。Navigation core 第一阶段可以继续以 topic 为主，但必须保留 action 兼容设计：

| Action | 来源 | 说明 |
|---|---|---|
| `get_plan` | `global_planner` | 普通全局规划请求 |
| `get_dwa_plan` | `dynamic_window_aware_global_planner` | 带动态窗口感知的全局规划请求 |

兼容要求：

- 不删除 action 能力。
- 允许上层继续通过 `/clicked_point` 触发规划。
- 允许第三方后端通过 action 请求路径。
- action 输出路径应与 `/global_path` 或 `/awared_global_path` 保持语义一致。

## 地形与地图保存接口

`terrain_analysis` 和 `save_terrain_map` 相关接口需要保留：

| Topic/Service | Type | 说明 |
|---|---|---|
| `/terrain_map` | `sensor_msgs/msg/PointCloud2` | 地形分析输出 |
| `/base_footprint_fill_cloud` | `sensor_msgs/msg/PointCloud2` | 机器人底盘附近补齐地面点 |
| `/save_terrain_map` | `std_srvs/srv/Trigger` | 触发保存地形地图 |
| `/map_clearing` | `std_msgs/msg/Float32` | 地图清理控制 |
| `/joy` | `sensor_msgs/msg/Joy` | 原 terrain_analysis 支持的手柄输入 |

Navigation 的地图保存封装必须覆盖自动保存和服务触发保存两种方式。

## 可选 SLAM 扩展服务

`FASTLIO2_ROS2-main` 提供 PGO、ICP 重定位和地图优化能力。它们不作为第一阶段主链路，但接口需要记录，后续可作为可选模块接入：

| Service | Type | 来源 | 说明 |
|---|---|---|---|
| `/pgo/save_maps` | `interface/srv/SaveMaps` | `pgo` | 保存回环优化后的地图和 patches |
| `/localizer/relocalize` | `interface/srv/Relocalize` | `localizer` | ICP/NDT 风格重定位请求 |
| `/localizer/relocalize_check` | `interface/srv/IsValid` | `localizer` | 查询重定位结果是否有效 |
| `/hba/refine_map` | `interface/srv/RefineMap` | `hba` | 一致性地图优化 |
| `/hba/save_poses` | `interface/srv/SavePoses` | `hba` | 保存优化后的位姿 |

第一阶段可以只在文档和配置中预留，不强制启动。
