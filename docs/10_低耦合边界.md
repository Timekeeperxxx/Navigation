# 10 低耦合边界

## 核心原则

Navigation 是独立导航系统，不是 BotDog 的子模块。

BotDog 只是当前已有前后端的参考接入方。后续其他 Python 后端、Web 后端、机器人调度系统、测试脚本，也应能通过同一套 Navigation 接口接入。

## 依赖方向

允许：

```text
上层应用 -> Navigation ROS2 topic/service/action
上层应用 -> Navigation CLI
上层应用 -> Navigation HTTP/WebSocket API
上层应用 -> Navigation 地图目录规范
```

禁止：

```text
Navigation core -> BotDog backend Python 模块
Navigation core -> BotDog 数据库
Navigation core -> BotDog 前端
Navigation core -> /home/jetson/Project/BOTDOG/... 固定路径
Navigation core -> BotDog data/nav_runtime 内部语义
```

## Navigation Core

核心层只负责导航本身：

```text
nav_bringup
  统一 launch
  统一 config
  启动 Livox / Super-LIO / terrain_analysis / global_planner

robot_nav_interfaces
  可选，自定义 msg/srv/action

robot_nav_runtime
  通用 runtime、pid、ready、日志管理

robot_nav_maps
  通用地图目录、metadata、PCD 文件约定
```

核心层不得 import 或调用 BotDog 代码。

## Adapter 层

所有和具体上层系统有关的兼容逻辑都放到 adapter：

```text
Navigation/adapters/
  legacy_scripts/
    scripts/
      start_mapping.sh
      restart_navigation_localization.sh
      stop_navigation.sh
    README.md
  botdog/
    README.md
```

legacy adapter 可以做：

- 生成上层后端常用的 ready 文件。
- 生成 PID 文件。
- 提供 `start_mapping.sh`、`restart_navigation_localization.sh`、`stop_navigation.sh` 兼容入口。
- 调用 Navigation 统一 launch。
- 输出通用 JSON 状态。

BotDog adapter 不可以做：

- 修改 Super-LIO 核心算法。
- 修改 global_planner 核心算法。
- 把 BotDog 业务状态写进 Navigation core。
- 要求 Navigation 必须运行在 BotDog 项目目录下。

## 公共接口优先级

接口设计优先级：

1. 标准 ROS2 topic/service/action。
2. 通用 CLI。
3. 通用 HTTP/WebSocket API。
4. legacy adapter 兼容脚本。
5. BotDog adapter 示例。

legacy 兼容脚本是最后一层，不是核心入口。

## 路径策略

Navigation core 只接受配置化路径：

```text
ROBOT_NAV_ROOT
ROBOT_NAV_MAP_ROOT
ROBOT_NAV_LOG_ROOT
ROBOT_NAV_RUNTIME_ROOT
SUPERLIO_WS
GLOBAL_PLANNER_WS
```

禁止在 core 中写死：

```text
/home/jetson/Project/BOTDOG
/home/jetson/superlio
/home/jetson/dddmr_navigation_new_local
/home/frankluo/Projects/BotDog-jetson-main
```

这些路径只能出现在本地 `.env`、YAML 配置、部署脚本或 adapter 示例里。

## Topic 策略

第一阶段可以继续使用 BotDog 已经适配的 topic 名：

```text
/clicked_point
goal_yaw
/nav_start
/nav_stop
/initialpose
/global_path
/nav_status
/lio/cloud_world
```

但文档和代码中应把它们定义为 Navigation public topics，而不是 BotDog topics。

后续如果新增命名空间，建议支持 remap：

```text
/nav/goal
/nav/start
/nav/stop
/nav/initialpose
/nav/global_path
/nav/status
```

并通过 launch remap 兼容旧 topic。

## 验收标准

低耦合验收必须满足：

- 删除或不启动 BotDog，Navigation 仍可独立建图、保存地图、重定位、规划。
- Navigation core 不 import BotDog Python 模块。
- Navigation core 不读取 BotDog 数据库。
- Navigation core 不要求地图在 BotDog 目录下。
- legacy adapter 可单独禁用。
- BotDog adapter 可单独禁用。
- 其他 Python 后端可以只用 ROS2 topic/service/action 接入。

## 测试前后端边界

Navigation 可以提供轻量测试前后端，但它属于开发验证工具：

```text
Navigation/tools/test_app/
  backend/
  frontend/
```

它可以调用 Navigation 公共接口，但 Navigation core 不得反向依赖它。
