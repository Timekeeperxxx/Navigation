# 09 实施计划

## 阶段 0：冻结现状

只读确认：

- BotDog 当前 API、topic、脚本调用方式。
- Super-LIO 建图和重定位入口。
- terrain_analysis 保存输出。
- global_planner launch 参数。

产物：

- 本文档集。
- 不改源码。

## 阶段 1：本地 bringup 封装

新增独立 Navigation 工作区，不改算法包核心。

建议新增：

```text
Navigation/
  src/
    nav_bringup/
  config/
  scripts/
  docs/
  maps/
  logs/
  runtime/
```

`nav_bringup` 提供：

- `mapping.launch.py`
- `navigation.launch.py`
- `localization.launch.py`
- `planning.launch.py`

同时建立示例功能覆盖矩阵，确保 `Superlio-main` 和 `dddmr_global_planner-main` 中的能力都被归类为：

- 核心必做。
- adapter 兼容。
- 可选扩展。
- 工具保留。
- 暂不纳入但记录原因。

## 阶段 2：通用兼容脚本

目标：

- 提供 `Navigation/adapters/legacy_scripts`。
- 上层后端可以继续用 subprocess 调脚本。
- BotDog 前端 API 不变。
- BotDog 后端 `services_ros_nav.py` 不大改。
- 只把 `start_mapping.sh` 和 `restart_navigation_localization.sh` 的底层启动逻辑切到统一 launch。

保留：

- PID 文件。
- ready 文件。
- 日志文件。
- PCD 查找兼容规则。

脚本对外形式：

```bash
bash start_mapping.sh <map_dir>
bash restart_navigation_localization.sh <scene_dir>
bash stop_navigation.sh
```

这些脚本不得依赖 BotDog 内部模块，只能依赖环境变量、参数和 Navigation 公共入口。

## 阶段 2.5：ROS2 运行时话题封装

目标：

- 新增 `nav_runtime` 包。
- 通过 `/nav/command_json` 和 `/nav_command_json` 接收 JSON 命令。
- 通过 `/nav/runtime_status` 发布运行时状态。
- 通过 `/nav/command_result` 发布命令结果。
- 底层仍调用 `Navigation/adapters/legacy_scripts`，保持 BotDog 和第三方后端都能接入。

第一批命令：

```text
start_mapping
restart_navigation
stop
status
```

该阶段只封装调度边界，不改 SLAM、重定位、规划算法。

## 阶段 3：轻量测试前后端

新增独立测试应用，用于验证 Navigation 公共接口，不依赖 BotDog。

目标：

- 用最小 FastAPI 后端调用 legacy scripts 和 ROS2 topic。
- 用最小 Web 前端触发建图、重定位、发送目标点、查看状态和路径。
- 作为 Navigation 自测工具，也作为第三方接入样例。

当前第一版已经落地到：

```text
Navigation/tools/test_app
```

实现范围：

- 标准库 HTTP 后端。
- rclpy topic 发布/订阅桥接。
- 兼容脚本进程管理。
- 单页 HTML 测试面板。
- Server-Sent Events 状态推送。

## 阶段 4：本地联调

验证：

- 建图保存。
- 地图加载。
- initialpose 重定位。
- clicked_point 规划。
- global_path 转发。
- BotDog 前端显示。
- 宇树控制链路不被破坏。

## 阶段 5：动态避障联调

目标：

- 接入实时点云或局部地图。
- 输出动态障碍状态。
- 对当前全局路径做碰撞监测。
- 近距离障碍触发停车、绕行或重规划。
- 保持全局路径规划、重定位和 BotDog/测试前端接口稳定。

第一阶段只做保守避障：

- 检测路径前方安全距离内是否有障碍。
- 有障碍时发布状态并限速/停车。
- 障碍持续存在时触发重规划请求。

不在第一阶段做复杂局部轨迹优化。

## 阶段 6：Jetson ARM64 原生迁移

目标：

- 不使用 Docker。
- 在 Jetson 上 colcon build。
- 路径只改配置，不改代码。
- Livox host IP 和网卡配置独立。

## 阶段 7：Docker 化

前提：

- 本地和 Jetson 原生均稳定。

目标：

- runtime 镜像。
- 可选 client/API 镜像。
- host 网络。
- volume 挂载地图和日志。
- BotDog 可继续通过 topic 和 API 访问。

## 代码改动纪律

- 先封装 launch、配置和脚本，不碰核心算法。
- 每次只改一个边界：建图、重定位、规划、BotDog 适配分开提交。
- 保留旧脚本备份或兼容入口，方便回滚。
- 所有硬编码路径必须被配置项替代。
- 每个阶段都要有可执行健康检查。
- 每次删减或暂缓示例工程功能时，都必须在覆盖矩阵里说明原因和替代方案。
