# 12 动态避障方案

## 目标

Navigation 需要支持动态避障，但不能破坏当前“先建图保存，再加载地图规划”的主流程。

动态避障应作为运行阶段的实时安全层，处理静态地图中没有的临时障碍，例如人、箱子、临时设备、移动物体等。

## 分层原则

动态避障不要直接混进静态全局规划器核心算法。

推荐拆成三层：

```text
全局规划层:
  使用 map.pcd / ground.pcd / footprint_fill.pcd
  负责从当前位置到目标点生成全局路径

动态感知层:
  使用实时 LiDAR 点云、terrain_analysis、local_map 或 perception layer
  负责识别机器人附近动态障碍和实时可通行区域

安全执行层:
  检查全局路径前方是否被动态障碍阻断
  必要时限速、停车、绕行或请求重规划
```

这样可以保持 global_planner 的静态地图能力稳定，同时给真实运行加上安全闭环。

## 推荐数据流

```text
Livox MID360
  -> /livox/lidar
  -> Super-LIO
  -> /lio/cloud_world
  -> 动态障碍提取
  -> /nav/local_obstacles
  -> 路径碰撞监测
  -> /nav/obstacle_status
  -> 停车/限速/重规划
```

如果使用 terrain_analysis：

```text
/lio/cloud_world + /lio/odom
  -> terrain_analysis
  -> /terrain_map
  -> 动态可通行区域/障碍判断
```

如果后续接入局部地图：

```text
/local_map_raw
  -> 动态障碍层
  -> /nav/local_obstacles
```

## 第一阶段能力

第一阶段只做保守动态避障，不做复杂轨迹优化。

当前第一阶段实现由三个节点组成：

| 节点 | 默认状态 | 作用 |
|---|---|---|
| `dynamic_avoidance_monitor.py` | 开启 | 裁剪 `/global_path` 为 `/prune_plan` 和 `/nav/local_path`，检测 `/nav/local_obstacles` 是否阻断路径，发布状态、重规划请求和 `/cmd_vel_safe` |
| `local_obstacle_simulator.py` | 关闭 | 没有雷达时通过 `/nav/sim_obstacle_point` 手动生成 `/nav/local_obstacles` |
| `nav_path_follower.py` | 关闭 | 轻量路径跟随器，根据 `/nav/local_path` 或 `/global_path` 生成保守 `/cmd_vel` |

必须实现：

- 订阅当前全局路径 `/global_path`。
- 获取机器人位姿 `map -> base_footprint`。
- 获取实时局部障碍点云。
- 检查路径前方一定距离内是否有障碍。
- 障碍进入安全半径时发布 `/nav/obstacle_status`。
- 障碍过近时发布停车或限速信号。
- 障碍持续阻断时发布 `/nav/replan_request`。

第一阶段不做：

- 不重写 Hybrid A*。
- 不实现复杂 MPC/TEB/DWA 局部规划。
- 不直接控制宇树 SDK 底层接口。
- 不把动态障碍永久写入静态地图。

## 公共 ROS2 接口

输出 topic：

| Topic | Type | 说明 |
|---|---|---|
| `/nav/local_obstacles` | `sensor_msgs/msg/PointCloud2` | 实时局部障碍点云 |
| `/nav/local_traversable` | `sensor_msgs/msg/PointCloud2` | 实时可通行点云，可选 |
| `/nav/obstacle_status` | `std_msgs/msg/String` JSON | 动态避障状态 |
| `/nav/replan_request` | `std_msgs/msg/Bool` | 请求全局重规划 |
| `/cmd_vel_safe` | `geometry_msgs/msg/Twist` | 可选，安全过滤后的速度 |

`/nav/obstacle_status` 示例：

```json
{
  "status": "blocked",
  "nearest_obstacle_distance": 0.72,
  "path_blocked": true,
  "action": "stop",
  "message": "路径前方存在动态障碍，已请求停车",
  "timestamp": 1770000000.123
}
```

状态建议：

- `clear`：路径前方安全。
- `caution`：附近有障碍，建议限速。
- `blocked`：路径被阻断，需要停车。
- `replan_requested`：障碍持续存在，已请求重规划。
- `sensor_lost`：实时点云或定位不可用。

## 安全策略

建议参数：

```yaml
dynamic_avoidance:
  enabled: true
  obstacle_topic: /nav/local_obstacles
  global_path_topic: /global_path
  robot_frame: base_footprint
  map_frame: map
  lookahead_distance: 2.0
  stop_distance: 0.6
  slow_distance: 1.2
  path_corridor_radius: 0.45
  replan_blocked_duration: 2.0
  sensor_timeout: 0.5
```

含义：

- `lookahead_distance`：只检查机器人前方一段路径。
- `path_corridor_radius`：路径走廊半径，障碍进入该区域视为可能阻断。
- `stop_distance`：障碍过近时停车。
- `slow_distance`：障碍较近时限速。
- `replan_blocked_duration`：障碍持续阻断多久后请求全局重规划。

## 与全局规划的关系

动态避障不是全局规划的替代。

全局规划仍负责：

- 根据静态地图生成全局路径。
- 当动态障碍持续阻断时重新规划。

动态避障负责：

- 实时监测路径安全。
- 对短时障碍停车等待。
- 对持续障碍请求重规划。
- 对控制命令做安全限幅。

## 与宇树控制的关系

动态避障层不直接依赖宇树 SDK。

推荐关系：

```text
/global_path
  -> 路径跟踪/控制桥
  -> /cmd_vel
  -> 安全过滤
  -> /cmd_vel_safe
  -> Unitree SDK bridge
```

如果当前系统已经有控制桥，也可以先让动态避障发布状态，由控制桥订阅后停车。

第一阶段优先使用状态信号和停车信号，避免直接改动宇树底层控制。

## 验收标准

动态避障第一阶段完成后，应满足：

1. 没有动态障碍时，不影响 `/global_path` 正常发布和执行。
2. 路径前方放置障碍时，`/nav/obstacle_status` 变为 `blocked`。
3. 障碍过近时，系统能触发停车或输出停车请求。
4. 移走障碍后，状态恢复 `clear`。
5. 障碍持续阻断超过阈值后，发布 `/nav/replan_request`。
6. 定位或点云超时时，状态变为 `sensor_lost`。
7. 不修改静态 `map.pcd`、`ground.pcd`、`footprint_fill.pcd`。
