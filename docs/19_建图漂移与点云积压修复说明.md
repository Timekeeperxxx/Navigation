# 建图漂移与点云积压修复说明（零基础版）

## 一句话概括

**以前是“点云和 IMU 在门口排队、正常帧又被误丢、滤波器还把自己的姿态尺子越用越歪”；现在通过接收与计算分线程、正确同步、修复 ESKF、用可靠地面约束横滚/俯仰，让数据及时进入、姿态不再慢慢倾斜，同时保留真实上下楼高度。**

---

## 1. 这份文档解决什么问题

现场建图出现过以下现象：

- 下午多次建图中，除个别场景外，地图都有不同程度漂移。
- `/livox/lidar` 正常应约每 `0.1 秒` 到一帧，但现场最大间隔达到 `0.3～0.4 秒`。
- RViz 一开始正常，运行越久延迟越大；最后转动雷达后要等 `4～5 秒` 才显示。
- Scene28、Scene31、Scene35 等地图出现“一边高、一边低”。
- Scene35 中原本相差约一层楼的区域一度只差约 `2 米`，继续下楼后又突然拉大到约 `7～10 米`。
- Scene37 在同一楼层实际走了约 `230 米`，轨迹的 Z 却累计下降约 `8 米`。
- 日志大量出现：

  ```text
  drop LiDAR frame: missing IMU at scan start
  ```

这不是一个单独参数造成的，而是“性能积压、时间同步、滤波器状态更新、俯仰缺少长期绝对参考”叠加后的结果。

---

## 2. 先用最简单的话解释建图过程

MID360 每秒大约产生：

- `10 帧`点云；
- `200 帧`IMU 数据。

可以把它想成拍照：

- 雷达点云是一张“全景照片”；
- IMU 记录拍这张照片时雷达怎么转、怎么移动；
- Super-LIO 用 IMU 把运动中的照片扶正，再和已有地图对齐。

一帧点云不是在同一瞬间拍完的。帧内每个点都有自己的采集时间，因此必须同时拿到覆盖整帧时间范围的 IMU 数据。如果 IMU 排队、缺失或时间边界判断错误，点云就可能被丢弃，或者去畸变错误。

地图高度还依赖姿态。假设系统把雷达俯仰角算错了 `2°`，在短距离内看不明显，但走 `230 米` 后会产生：

```text
230 × tan(2°) ≈ 8 米
```

这正好与 Scene37 的“同层走 230 米，Z 下降约 8 米”处于同一数量级。也就是说，地图看起来像一直下坡，并不一定是系统真的认为机器人垂直下降，也可能是它使用的“水平尺”歪了。

## 3. 根因一：传感器回调被建图计算堵住

### 原来是什么情况

原来的 IMU 回调、LiDAR 回调和点云匹配：

- 使用同一个互斥回调组；
- 使用单线程 `spin()`；
- 点云匹配、地图更新稍慢时，IMU 和下一帧点云只能在后面排队。

这相当于只有一个收费窗口：前面有一辆大货车办理复杂业务，后面的救护车也只能等。

IMU 约 `200 Hz`，原驱动内部队列只有 `64 条`，只能容纳约：

```text
64 ÷ 200 = 0.32 秒
```

Jetson 一旦卡顿超过这个时间，就可能丢旧 IMU。RViz 最后延迟 `4～5 秒`，也是处理速度长期低于数据产生速度的典型表现。

### 现在怎么改

- 传感器接收和 LIO 计算使用两个独立回调组。
- 使用两线程 `MultiThreadedExecutor`：
  - 一个线程及时收 IMU/LiDAR；
  - 一个线程做去畸变、匹配和地图更新。
- IMU 回调只负责校验时间并放入缓存，不再直接修改 ESKF 或发布两套高频预测里程计。
- 缓存只在“放入数据、复制一帧测量”时短暂加锁；点云匹配期间不持锁。
- 驱动 IMU 队列从 `64` 扩大到 `512`，约能吸收 `2.56 秒`的临时调度抖动。
- 应用层 IMU 缓存最多保留约 `3 秒`，防止断开雷达后无限增长。
- LiDAR 队列保持很小，实时点云 QoS 使用 `KeepLast(1)`，避免 RViz 慢时堆积过期大点云。
- FastDDS 大消息发布改为异步，发布点云时不长时间堵住 ROS 执行器。

### 为什么不能只把所有队列无限加大

因为队列越大不等于越实时。无限队列只会把“现在看到的画面”变成“几秒前的历史画面”，最终内存也会被吃完。

正确做法是：

- IMU 很小、频率高，保留几秒有意义；
- 点云很大，只保留最新少量帧；
- 从结构上让接收不再被重计算堵住。

## 4. 根因二：正常的 Livox 帧边界抖动被当成缺失

### 原来是什么情况

Livox 相邻点云帧的边界并不保证恰好无缝，有时会：

- 重叠约 `0.19～0.26 毫秒`；
- 间隔约 `0.45～0.58 毫秒`。

这是正常组帧抖动。但原同步逻辑只容许约 `1 微秒`，也就是：

```text
正常抖动：约 200～580 微秒
原来容差：约 1 微秒
```

尺子比实际需求严了几百倍，因此大量正常帧被错误判为：

```text
missing IMU at scan start
```

Scene37 原运行的同步丢弃率一度长期达到 `40%～50%`，累计误丢约 `2290 帧`。

### 现在怎么改

- 扫描边界容差改为可配置，默认 `1 毫秒`。
- 扫描开始必须有之前或容差内的 IMU 样本。
- 扫描结束使用前后两条 IMU 做线性插值，生成时间完全对应的边界状态。
- 下一帧继续复用这个精确边界，不因为一帧点云异常而顺带丢掉中间有效 IMU。
- 扫描内部任何 IMU 积分间隔超过 `50 毫秒`，仍然判定为真实不连续。
- 只放宽正常边界抖动，绝没有把“缺多少都算正常”。

### 实际效果

同一 Scene37 录包：

- 修复前实际进入 LIO 的点云约 `3441 帧`；
- 修复后约 `5500 帧`进入 LIO；
- 原来反复出现的 `missing IMU at scan start` 和 `40%～50% synchronization unhealthy` 消失。

不过，修复同步后终点 Z 仍约为 `-7.62 米`。这说明同步是一个必须修的问题，但不是 Scene37 高度斜坡的唯一根因。

## 5. 根因三：点云帧内去畸变使用的时间不够准确

一帧点云中的每个点采集时间不同。原平移补偿部分可能使用统一的 IMU 时间间隔，没有完全使用该点自己的查询时间。

现在开启：

```yaml
lio.sensor.use_query_time_undistort: true
```

每个点使用自己的真实时间偏移计算旋转和位移，降低运动时的墙面拉伸、重影以及错误平面方向。

## 6. 根因四：ESKF 协方差更新把“没测到的状态”当成测到了

### 先解释 ESKF 是什么

ESKF 可以理解成一个不断更新的“机器人状态账本”，里面记录：

- 姿态；
- 位置；
- 速度；
- 陀螺仪零偏；
- 加速度计零偏；
- 重力状态。

它还记录自己对每一项有多大把握，这个“把握程度”就是协方差。

### 原来的错误

激光匹配直接观察的是六维位姿，不会直接测到所有速度、IMU 零偏和重力状态。

原协方差更新临时矩阵使用了单位阵，相当于告诉滤波器：

> “这些没有被激光测到的状态也已经测得很准。”

这会反复压小相关协方差，让滤波器变得过度自信。之后 IMU 零偏慢慢变化时，系统也不愿意调整，误差更容易进入速度和高度。

### 现在怎么改

临时矩阵从 `Identity()` 改为 `Zero()`，只把真正观测到的六维位姿块填进去。

这样速度、IMU 零偏和重力等未直接观测状态保留合理的不确定性，滤波器能够继续正常估计它们。

## 7. 根因五：长距离行走时，横滚/俯仰缺少可靠的绝对参考

### 为什么单纯切换 `estimate_gravity` 没解决

当前配置是：

```yaml
lio.kf.kf_align_gravity: true
lio.kf.estimate_gravity: false
```

含义是：

- 启动时仍使用静止 IMU 对齐重力；
- 初始化后不把重力方向作为 ESKF 状态持续在线修改；
- ESKF 预测仍然正常使用重力，并不是“关闭重力补偿”。

同一 Scene37 包的 A/B 结果：

```text
estimate_gravity=false：终点 Z ≈ -7.623 m
estimate_gravity=true： 终点 Z ≈ -7.555 m
```

只改善约 `7 厘米`，对 `7～8 米`漂移可以视为无效。因此不能把开关改回 `true` 就宣布问题解决。

### 现在新增什么约束

新增“经过 IMU 复核的局部水平面姿态约束”：

1. 统计最近约 `1 秒`、加速度模长正常的 IMU 数据，得到大致竖直方向。
2. 从当前点云中选择雷达下方 `0.05～3 米`、距离不超过 `18 米`的点。
3. 使用 RANSAC 拟合局部平面。
4. 至少满足以下条件才采用：
   - 至少 `40` 个平面内点；
   - 内点比例至少 `12%`；
   - 平面点误差阈值约 `6 厘米`；
   - 平面法向与 IMU 竖直方向夹角不超过 `3°`；
   - 对当前姿态的修正量不超过 `6°`。
5. 条件不满足时，本帧直接放弃该约束，继续使用正常 LIO。

### 它到底约束什么

它只给横滚和俯仰提供绝对参考，不约束：

- X；
- Y；
- Z；
- 偏航角；
- 竖直速度。

因此这不是“把 Z 锁在零附近”，也不是“强行让机器人永远在同一层”。

### 为什么能解决同层 Z 斜坡

同层长距离出现 Z 斜坡，常常是俯仰角慢慢偏了。局部地面相当于不断给系统看一把真实水平尺，让俯仰不能在弱约束方向上越走越歪。

### 上下楼和坡道怎么办

- 楼梯踏步通常仍是局部水平面，约束姿态但不锁高度，所以 Z 可以一级一级正常变化。
- 明显坡道的平面法向与重力方向夹角超过阈值时，该平面约束会被拒绝，不会强行把坡道压平。
- 没有足够可靠地面点时也会自动拒绝。

---

## 8. 为什么没有使用“强行锁 Z”的方案

曾考虑过同层高度保护带，例如把 Z 限制在某个范围。这种办法看起来立刻有效，但有严重问题：

- 真下楼时也可能被拦住；
- 商场多层地图会被压在一起；
- 只能遮住结果，不能修正导致地图倾斜的姿态原因。

最终代码没有使用固定 Z 保护带，而是修正“水平尺”，让真实高度仍由 IMU 和点云匹配共同估计。

同样，也没有采用强权重的动态加速度校平。机器人加速、减速、转弯时，加速度计读数不全是重力；强行把瞬时加速度当成重力会破坏 XY 轨迹。

## 9. 其他可靠性和性能修改

### 9.1 更严格的初始化

启动时使用约 `400` 个 IMU 样本，并检查：

- 角速度大小；
- 角速度波动；
- 加速度波动。

只有确认设备静止才完成初始化。现场操作要求：

1. 把机器人放在平整地面；
2. 雷达保持正常安装姿态；
3. 启动后至少静止 `5 秒`；
4. 初始化完成前不要把横放的雷达再转成竖放。

如果初始化时横放、完成后再竖起来，世界重力不会神奇地重新定义，初始姿态和后续状态会产生很大突变，极易造成地图异常。

### 9.2 拒绝危险的单帧匹配结果

新增基本安全检查：

- 有效匹配点不能太少；
- 单帧平移不能异常过大；
- 单帧旋转不能异常过大；
- 位姿必须是有限数值。

异常时不自动暂停建图，而是拒绝这一次不可信的地图更新，回退到 IMU 预测或上一可靠状态。符合现场要求：**可以告警，但不能因为短暂不同步自动暂停整次建图。**

### 9.3 地图预览不再反复过滤整张大地图

原来每隔若干帧可能重新处理不断增长的整张预览地图，建图越久计算量越大。

现在：

- 增量维护预览点云；
- 默认每 `20 帧`发布一次预览；
- 预览最多约 `15 万点`；
- 超过上限自动增大预览体素；
- 最终 `map.pcd` 的完整保存数据与 RViz 预览分开；
- 增加平均、P99、最大耗时和慢阶段日志。

这里的“体素滤波”可以理解为：把空间划成许多小方格，每格保留一个代表点。它的目的是减少重复点和显示负担，不是删除整片真实结构。预览可以稀一点，最终地图仍使用自己的保存数据和分辨率。

### 9.4 停止建图顺序和地图保存

正常停止顺序现在是：

1. 调用 `/lio/pause_mapping` 冻结 Super-LIO 输入并清空尚未处理的传感器队列；
2. 让 Super-LIO 正常退出并保存最终 `map.pcd`；
3. 调用 `/save_terrain_map` 保存 terrain、ground 和 footprint 点云；
4. 再关闭其余节点。

这样避免用户点击停止后，地图还继续变化，导致主地图和地面地图不是同一最终状态。

PCD 保存改成：先写 `.tmp` 临时文件，成功后再重命名成正式文件。即使写盘中途失败，也不会留下一个看起来正常、实际损坏的正式 PCD。

`ground` 显式保存成功后，退出时不会再重复进行一次耗时过滤和写盘。

### 9.5 长地图地面过滤加速

地面保存原来可能对大量点反复搜索邻域，地图越大越慢。现在先按 XY 和 Z 网格聚合，再对少量高度层做邻域统计，把近似的“逐点对大量邻点”改为“点只入桶一次，再比较桶”。

这不会改变保存目标，主要作用是避免长地图结束时地面过滤耗时过长。

## 10. 雷达前倾 19.48° 会不会导致漂移

雷达前倾本身不会必然导致漂移。它是固定安装姿态，目的是看到更多地面。只要：

- 安装角测量正确；
- `NAV_LIDAR_MOUNT_PITCH_DEG=19.48` 与实物一致；
- 初始化期间雷达保持正常安装姿态；
- `base_link`、`base_footprint` 变换方向正确；

系统可以正确处理。

`19.48°` 描述的是“雷达相对机器人”的安装角，不是世界地面应该倾斜 `19.48°`。世界竖直方向由初始化和姿态估计确定，两者不能混为一谈。

如果实际安装角不是 `19.48°`，需要重新测量并修改后台参数；不要通过调整重力方向去补偿错误的机械外参。

## 11. `ground.pcd` 为什么有时不存在

单独执行：

```bash
ros2 run nav_lio super_lio_node ...
```

只会运行 LIO，它负责保存 `map.pcd`，不会自动提取地面。

`ground` 需要同时运行：

- `nav_terrain_analysis`：识别地面；
- `nav_save_terrain_map`：累计并保存地面。

因此录包验证应使用完整的 `mapping.launch.py`，并设置 `launch_terrain:=true`。

地面文件通常不是字面上的 `ground.pcd`，而是：

```text
terrain_map_YYYYMMDD_HHMMSS_ground.pcd
```

## 14. 怎么判断下一次测试是否正常

### RViz 观察

- 转动雷达后点云应及时响应，不能延迟越来越大。
- 同一面墙多次经过不应明显重影或逐渐倾斜。
- 同一楼层长距离行走，地面不应形成持续单向斜坡。
- 上下楼时应出现真实、连续的楼层高差，不应先压在一起再突然拉开。

### 日志观察

正常目标：

- `Livox IMU publish queue overloaded` 为零或极少；
- 初始化完成后，`missing IMU at scan start` 为零；
- 不持续出现 `processing backlog`；
- `level constraint health` 中有稳定的 accepted 计数；
- `post_update_innovation` 保持较小；
- 不持续出现 `reject unsafe map update`。

偶尔一条告警不一定表示失败；需要重点看是否持续出现、计数是否快速增长、RViz 延迟是否不断累积。

### 文件观察

正常结束后场景目录至少应看到：

```text
map.pcd
terrain_map_时间_ground.pcd
```

## 15. 当前关键参数

```yaml
lio.sensor.use_query_time_undistort: true
lio.sensor.max_imu_integration_dt: 0.05
lio.sensor.scan_boundary_tolerance: 0.001

lio.kf.kf_align_gravity: true
lio.kf.estimate_gravity: false

lio.kf.level_constraint.enable: true
lio.kf.level_constraint.gravity_window_sec: 1.0
lio.kf.level_constraint.max_plane_gravity_angle_deg: 3.0
lio.kf.level_constraint.max_attitude_innovation_deg: 6.0

lio.output.map_preview_publish_interval: 20
lio.output.map_preview_max_points: 150000
```

不要只看参数名直接修改。尤其是：

- `estimate_gravity=false` 不等于不使用重力；
- `scan_boundary_tolerance=0.001` 的单位是秒，也就是 `1 毫秒`；
- 地面姿态约束不等于锁定 Z；
- 雷达安装俯仰角不等于世界地面角度。

## 17. 最后再用一句大白话总结

**以前的问题就像“摄像头画面堵在队列里、时间表又把正常照片扔了，最后机器人拿着一把歪了 2° 的水平尺走了 230 米”；现在先保证数据及时、完整、时间正确，再修好滤波器，并用可信地面不断校验水平尺，所以地图既不会越跑越延迟，也不容易把同一层楼画成 8 米斜坡。**
